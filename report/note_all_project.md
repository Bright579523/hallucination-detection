# 🛡️ Defense Questions: Engineering Rationale
(คำถามคลาสสิกสไตล์กรรมการสอบวิทยานิพนธ์ และแนวทางการตอบ)

นี่คือเหตุผลแบบเจาะลึกที่คุณสามารถนำไปใช้ตอบกรรมการหรือเขียนลงใน Report ได้อย่างมั่นใจครับ:

### 1. ทำไมต้องใช้ชุดข้อมูล COCO (Common Objects in Context)?
ไม่ได้เป็นเพราะมันแค่หาโหลดง่ายครับ แต่มีเหตุผลเชิงวิชาการที่แข็งแกร่งรองรับ:
* **มาตรฐานระดับโลก (Gold Standard):** ในวงการ Computer Vision COCO คือชุดข้อมูลมาตรฐานที่งานวิจัยระดับ Top-tier เกือบทุกชิ้นใช้ประเมินผล การที่เราใช้ COCO ทำให้ผลลัพธ์ของโปรเจกต์คุณมีความน่าเชื่อถือและสามารถนำไปเปรียบเทียบกับ Paper อื่นๆ ได้
* **ฟีเจอร์ "Supercategory" (อาวุธลับของโปรเจกต์นี้):** นี่คือเหตุผลสำคัญที่สุด COCO มีการจัดลำดับชั้นข้อมูล (Hierarchy) ไว้แล้ว เช่น หมวดหมู่สัตว์ (Animal) จะครอบคลุมทั้ง สุนัข แมว ม้าลาย ฯลฯ ทำให้คุณสามารถเขียนโค้ดสุ่ม **"Hard Negatives"** (เช่น เอารูปสุนัข มาหลอกโมเดลด้วยคำว่า หมาป่า) ได้แบบอัตโนมัติ โดยไม่ต้องไปนั่งหาข้อมูลหรือสร้าง Labels ใหม่เองเลย 

### 2. ทำไมต้องใช้โมเดลสำเร็จรูป OWL-ViT และ CLIP?
ในโปรเจกต์นี้เราต้องการสร้าง "ระบบตรวจสอบ (Verifier)" ไม่ใช่การสร้างโมเดลตรวจจับวัตถุแข่งกับ Google หรือ OpenAI การใช้ Pre-trained models จึงเป็นตัวเลือกที่ฉลาดที่สุดด้วยเหตุผลดังนี้:

* **OWL-ViT (The Target):** โมเดลเก่าๆ อย่าง YOLO หรือ Faster R-CNN จะตรวจจับได้แค่คลาสที่ถูกเทรนมาตายตัว (เช่น 80 คลาส) แต่ OWL-ViT เป็นโมเดล **Open-Vocabulary** ที่รับคำสั่งเป็นข้อความ (Text Prompt) อะไรก็ได้ให้ไปหาในภาพ ซึ่งความยืดหยุ่นตรงนี้นี่แหละที่ทำให้เกิด **"จุดอ่อนเรื่องการอุปโลกน์ (Hallucination)"** เราจึงต้องเลือกโมเดลประเภทนี้มาเป็นเป้าหมายในการทดสอบ
* **CLIP (The Independent Judge):** เราต้องการเครื่องมือมาช่วยเช็กผลลัพธ์ของ OWL-ViT แบบเป็นกลาง CLIP ถูกเทรนมาด้วยภาพและข้อความหลายร้อยล้านคู่เพื่อทำความเข้าใจ "ความหมาย (Semantics)" มันจึงทำหน้าที่เหมือน **กรรมการอิสระ** ที่คอยให้คะแนน (Cosine Similarity) ว่าสิ่งที่ OWL-ViT วงกรอบมาให้ กับข้อความที่ผู้ใช้พิมพ์ลงไป มันมีความสอดคล้องกันจริงๆ หรือแค่หลอนไปเอง

### 3. ข้อดีของ Google Colab
การเลือกใช้ Colab ตอบโจทย์การทำงานกลุ่มและข้อจำกัดทางทรัพยากรได้สมบูรณ์แบบ:
* **Collaboration (การทำงานร่วมกัน):** ตัดปัญหาเรื่อง "โค้ดรันผ่านในเครื่องฉัน แต่พังในเครื่องเพื่อน" (Dependency Hell) เพราะทุกคนทำงานบน Environment เดียวกันบนเบราว์เซอร์ และดึงไฟล์ภาพจาก Google Drive ก้อนเดียวกัน
* **Free GPU Power:** การประมวลผล Vision Transformer (ViT) ทั้งฝั่งภาพและข้อความบนรูปภาพ 5,000 รูปนั้น กินพลังการประมวลผลมหาศาล หากรันด้วย CPU ของแล็ปท็อปปกติอาจใช้เวลาเป็นวันๆ แต่ T4 GPU ฟรีของ Colab สามารถย่นเวลาให้จบได้ในหลักนาทีหรือชั่วโมงต้นๆ เท่านั้น

> **สรุป:** การผสมผสานระหว่างชุดข้อมูลที่มี Supercategory, โครงสร้างโมเดลแบบ Open-vocabulary, และแพลตฟอร์ม Cloud Computing ทำให้โปรเจกต์นี้ดูเป็น **Production-ready pipeline** ที่วิศวกรข้อมูลในบริษัท Tech ชั้นนำใช้งานกันจริงๆ ครับ

## ข้อมูลเชิงลึก (Deep-dive Insights) สำหรับบทวิเคราะห์ (Discussion)

> **💡 คำชี้แจงเพื่อความเข้าใจที่ตรงกันใน Report:**
> เราจะแบ่งระบบตรวจสอบ (Verifier) ออกเป็น 2 เวอร์ชั่นหลัก เพื่อใช้ในการเปรียบเทียบผลลัพธ์:
> * **V1 (Confidence Threshold Baseline):** คือระบบแบบพื้นฐาน ที่ตัดสิน Hallucination โดยดูแค่ "คะแนนความมั่นใจ (Confidence Score)" ของ OWL-ViT ถ้ามั่นใจน้อย ถือว่าหลอน (ตัวนี้คือ Baseline)
> * **V2 (CLIP Similarity Verifier):** คือระบบที่ฉลาดขึ้น โดยการนำภาพที่ OWL-ViT จับได้ มาครอบตัด (Crop) แล้วส่งให้โมเดล CLIP ประเมินความหมายซ้ำอีกรอบ (ตัวนี้คือโมเดลพระเอกที่เราเสนอ)

### 1. ปัญหาคอขวดตอนครอบตัดภาพ (The Cropping Resolution Problem)
* **สิ่งที่มักจะพลาดกัน:** ในขั้นตอนการทำงานของ **V2 (CLIP Verifier)** คุณต้องเอากรอบ Bounding Box ที่ OWL-ViT ตีไว้ ไปครอบตัด (Crop) รูปภาพ แล้วส่งต่อให้ CLIP ประเมิน
* **ข้อมูลเชิงลึก:** โมเดล CLIP มักจะต้องการรูปภาพขนาด 224x224 พิกเซล หาก OWL-ViT ตรวจจับ "นก" ที่อยู่ไกลมากๆ (Bounding box มีขนาดแค่ 20x20 พิกเซล) เมื่อคุณครอบตัดและขยายภาพ (Resize) ส่งให้ CLIP ภาพจะแตกและเบลอหนักมาก จน CLIP อาจจะให้คะแนน Cosine Similarity ต่ำ
* **สิ่งที่จะได้คะแนน:** ในส่วนวิเคราะห์ Error Analysis ให้คุณหยิบรูปที่ภาพแตกแบบนี้มาโชว์ แล้วอธิบายว่า *"นี่ไม่ใช่ Hallucination แต่เป็นข้อจำกัดด้าน Resolution ของ Image Pipeline"* กรรมการจะทึ่งในความละเอียดของคุณครับ

### 2. ปรัชญาของ AI: เส้นแบ่งระหว่าง "Hallucination" กับ "Misclassification"
* **สิ่งที่มักจะพลาดกัน:** การเหมารวมว่าทายผิดคือหลอนทั้งหมด
* **ข้อมูลเชิงลึก:** ในโมเดลทั่วไป การเห็น "สุนัขไซบีเรียนฮัสกี้" แล้วทายว่าเป็น "หมาป่า" คือการทายผิดหมวดหมู่ (Misclassification) แต่สำหรับโมเดล Open-Vocabulary ที่รับคำสั่ง Prompt การที่โมเดลพยายาม "ยัดเยียด" ให้ไซบีเรียนเป็นหมาป่าเพียงเพราะเราพิมพ์คำว่าหมาป่าลงไป ถือเป็น Hallucination แบบหนึ่งหรือไม่?
* **สิ่งที่จะได้คะแนน:** การเขียนถกเถียงประเด็นนี้ในบท Literature Review จะยกระดับ Report ของคุณให้ดูมีความเป็นงานวิจัยระดับสูง (Research Maturity) เพราะมันคือปัญหาที่วงการ AI กำลังถกเถียงกันอยู่จริงๆ

### 3. ทำไม ROC-AUC ถึงเป็นพระเอกตัวจริง (The Power of ROC-AUC)
* **สิ่งที่มักจะพลาดกัน:** คนมักจะโฟกัสแต่ค่า F1-Score หรือ Accuracy
* **ข้อมูลเชิงลึก:** F1-Score และ Accuracy จะเปลี่ยนไปตามค่า Threshold (เช่น ถ้าตั้ง 0.5 ได้ F1 เท่านี้ ถ้าตั้ง 0.6 F1 จะเปลี่ยนไป) แต่ **ROC-AUC** เป็นตัวชี้วัดที่เป็นอิสระจาก Threshold (Threshold-invariant) มันวัด "ความสามารถในการแยกแยะ (Separability)" ของโมเดลโดยรวม
* **สิ่งที่จะได้คะแนน:** ตอนเขียนผลลัพธ์ ให้เน้นอธิบายว่าโมเดล **V2 (CLIP)** มีค่า ROC-AUC สูงกว่า **V1 (Confidence Baseline)** อย่างมีนัยสำคัญ ซึ่งแปลว่าโดยเนื้อแท้แล้ว V2 มีศักยภาพในการแยกแยะ "ของจริง" กับ "ของปลอม (Hallucination)" ได้ดีกว่า V1 มาก เพียงแค่เราต้องหาค่า Threshold ที่เหมาะสมให้มันให้เจอ

### 4. ทำไมยังใช้โมเดล "รุ่นพี่" อย่าง OWL-ViT และ CLIP? (Rationale for Legacy Models)
หากกรรมการถามว่า *"ปัจจุบันมีโมเดลใหม่ๆ ที่แม่นกว่า ทำไมยังใช้ของเมื่อ 4-5 ปีที่แล้ว?"* นี่คือเหตุผลทางวิศวกรรมที่ใช้ตอบได้ทันทีครับ:

* **เราต้องการศึกษา "อาการหลอน" ไม่ใช่หา "คนสุขภาพดีที่สุด":** เป้าหมายของวิจัยนี้คือการสร้าง **"เครื่องจับเท็จ (Verifier)"** ดังนั้นเราจึงต้องการโมเดลตั้งต้น (OWL-ViT) ที่ยังมีจุดอ่อนและแสดงอาการ Hallucination ออกมาให้เห็นชัดเจน หากใช้โมเดลใหม่ที่แม่นยำ 99% เราก็จะไม่มีข้อมูลความผิดพลาดเหลือให้วิเคราะห์และทดสอบระบบ V2 ของเราเลย
* **ความโปร่งใส (White-box) vs กล่องดำ (Black-box):** ระบบ V2 ของเราต้องการมุดเข้าไปดึง **Embeddings (เวกเตอร์ตัวเลข 512 มิติ)** ออกมาคำนวณ Cosine Similarity ได้ตรงๆ ซึ่งโมเดล Open-source รุ่นพี่เหล่านี้ยอมให้เราทำได้ แต่โมเดล API รุ่นใหม่ (เช่น GPT-4o) มักเป็น "กล่องดำ" ที่ให้แค่คำตอบ แต่ไม่ยอมให้เราเข้าถึงค่าคณิตศาสตร์เชิงลึกข้างในได้
* **ความคุ้มค่าและเป็นไปได้ (Hardware Feasibility):** OWL-ViT และ CLIP มีขนาดพอเหมาะกับการรันบน **T4 GPU ฟรีของ Google Colab** (VRAM 16GB) ทำให้ใครก็ตามสามารถนำงานวิจัยนี้ไปทำซ้ำ (Reproduce) ได้จริง 100% โดยไม่มีค่าใช้จ่าย ในขณะที่โมเดลรุ่นใหม่ขนาดใหญ่ (เช่น LLaVA หรือ Grounding DINO รุ่นท็อป) มักต้องการ VRAM ระดับ 24GB-80GB ซึ่งต้องใช้การ์ดจอราคาแพงลิ่วในการรัน
* **เสถียรภาพในการวิจัย:** การใช้โมเดลที่ Weights ถูกเปิดเผยและนิ่งแล้ว ทำให้ผลลัพธ์การทดลองของเรามีความเสถียร ไม่เปลี่ยนไปมาเหมือนการเรียกใช้ API ของค่ายใหญ่ที่มักจะมีการอัปเดตเวอร์ชัน (Model Drift) อยู่ตลอดเวลา
